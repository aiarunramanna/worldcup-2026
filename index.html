    Full BMAD Pipeline Guide

    One heads-up before we start: exact command names shift a little between BMAD versions (*agent pm vs /pm vs a skill name like bmad-agent-pm). Whatever you have installed, bmad-help will always tell you the exact invocation — I'll use the general skill-name convention below, but treat bmad-help as your source of truth if something doesn't respond.

    Phase 0 — Install BMAD into this project

    BMAD installs per-project, so if this World Cup folder isn't already inside a BMAD-enabled repo:

    cd C:\tbizapp
npx bmad-method install

    The installer will ask:

    • Which modules? → choose bmm (BMad Method core — the agile agents)
    • Which IDE/tool? → claude-code
    • Project type → generic/static site is fine

    This drops a .bmad-core (or .claude/skills) folder into your project with all the agent definitions. Restart Claude Code afterward so it picks up the new skills.

    Phase 1 — Analysis (Analyst agent, "Mary")

    Invoke her:

    bmad-agent-analyst

    This is the step Quick Flow skips entirely — worth doing here just to see it. Mary will interview you rather than just take orders. Expect questions like: Who's this page for? What's the success criteria? What's out of scope? Answer honestly even though it's a small project:

    • Purpose: a page you can point people to, and a learning exercise in BMAD
    • Audience: you + anyone you send the link to
    • Constraints: static only, must live at aiarunramanna.github.io/worldcup-2026/, no backend, no build step
    • Success: correct schedule info, loads fast, readable on mobile

    Output: docs/project-brief.md — a short document capturing all of that. This becomes the input to everything downstream.

    Phase 2 — Planning (PM agent "John", optionally UX agent "Sally")

    bmad-agent-pm

    John reads project-brief.md and turns it into a PRD (docs/prd.md) — this is the artifact BMAD cares most about getting right. Expect it to contain:

    • Epic 1: Display Match Schedule — stories like "show semifinal 1 details," "show semifinal 2 details," "show final details," each with Given/When/Then acceptance criteria
    • Epic 2: Publish & Host — story for getting it live under the existing repo
    • Non-functional requirements: mobile-responsive, loads without JS errors, accurate as of build date

    Since this project has a UI, do the optional UX step too — it's part of full ceremony:

    bmad-agent-ux

    Sally produces docs/ux-spec.md: layout (single-column card per match), typography, color choices, mobile breakpoint behavior. For something this small it'll be short, but you'll see how she uses the PRD as her input.

    Phase 3 — Solutioning (Architect "Winston", then PO for sharding)

    bmad-agent-architect

    Winston reads the PRD + UX spec and makes the technical calls, documented as ADRs in docs/architecture.md. For this project expect something like:

    • ADR-001: No JS framework — static single-page, plain HTML/CSS, since PRD specifies no backend and no build step
    • File structure: worldcup-2026/index.html, inline <style> (or a co-located style.css)
    • Deployment target: subfolder of the existing aiarunramanna.github.io repo (already Pages-enabled, no new config needed)

    Then the PO agent shards the PRD + architecture into implementation-ready files and checks they're actually ready to build from:

    bmad-agent-po

    This produces docs/epics/epic-1-match-schedule.md and docs/epics/epic-2-publish-hosting.md, and runs a Definition-of-Ready check — does every story have clear acceptance criteria and an unambiguous architecture to build against? If something's fuzzy, the PO kicks it back rather than letting it into implementation half-specified. This is the checkpoint that's supposed to prevent the "Dev agent guesses wrong" problem.

    Phase 4 — Implementation loop (SM → Dev → QA, per story)

    This is the part that repeats — for each story, one at a time:

    1. Scrum Master creates the story 
      bmad-agent-sm

      Invoke something like *create-story against Epic 1's first story. The SM (traditionally "Bob") writes a self-contained story file — docs/stories/story-1.1-display-schedule.md — with the acceptance criteria and relevant architecture notes copied into the file, so the Dev agent doesn't need the whole project history in context.

    2. Dev implements — in a fresh chat

      Best practice here (and the whole point of story files): start a new Claude Code session so Dev isn't carrying baggage from the Analyst/PM/Architect conversation.

      bmad-agent-dev
*dev-story docs/stories/story-1.1-display-schedule.md

      Amelia reads only that story file, implements it (writes/edits worldcup-2026/index.html), and checks her own work against the acceptance criteria before handing back.

    3. QA reviews

      Reviews the diff against the story's acceptance criteria and the PRD's non-functional requirements — accurate match info, mobile-responsive, no console errors. Comes back with a gate result: PASS, or FAIL with specific notes. FAIL sends it back to Dev; PASS means you commit:

      git add worldcup-2026
git commit -m "Story 1.1: display match schedule"
    4. Repeat

      Repeat for story 1.2 ("News Flash" note), 1.3 (final match details), etc., then move to Epic 2's story (subfolder → push → verify live under GitHub Pages).

    Optional: Party Mode

    Worth trying once, since it's a distinctive BMAD feature: bmad-party-mode puts multiple agents in one session so they debate a decision together — e.g., pull in Architect + PM to argue out whether the ADR above (no JS framework) is right, instead of you relaying between them manually.
